# Техническое задание на разработку SEO-ориентированной PHP CMS

## Контекст и цели продукта

Разрабатываемая CMS предназначена для установки на обычный веб-хостинг (shared/VPS) с PHP и БД и должна «из коробки» обеспечивать публикацию SEO-ориентированных страниц и блог‑постов с высокой производительностью, предсказуемой индексацией и удобным редакторским опытом.

Ключевая продуктовая идея: объединить «простоту CMS для контента» с «SEO‑дисциплиной по умолчанию» и архитектурными решениями, которые минимизируют время ответа и стоимость рендеринга страниц (кэширование, предрендер, строгая модель блоков, защищённая работа с пользовательским HTML). Цель — по возможностям приблизиться к WordPress/Ghost‑классу, но сфокусироваться на: (а) SEO‑контролируемых страницах и листингах, (б) скорости и предсказуемой индексации, (в) API‑first публикации, (г) встроенной интеграции с LLM при сохранении безопасности и управляемости.

SEO-ориентированность в ТЗ опирается на требования поисковых систем к индексации/дедупликации, мета‑директивам, sitemap/robots и разметке structured data. Например, Google описывает каноникализацию как процесс выбора представительного URL среди дублей и подчёркивает важность корректного задания preferred canonical (через rel=canonical и другие сигналы). citeturn0search4turn0search0

Важная практическая цель — обеспечить «правильные» технические артефакты SEO по умолчанию: корректные canonical URL, robots meta/X‑Robots‑Tag, sitemap(ы), robots.txt на корне, structured data (желательно JSON‑LD), а также производительность, соответствующую ожиданиям по Core Web Vitals, которые Google прямо рекомендует поддерживать в «хорошей» зоне для успеха в поиске и UX. citeturn7search0turn7search5

## Пользовательские роли и сценарии

В системе должны существовать роли и права (RBAC) с серверной проверкой на каждом действии (не «на UI», а на уровне контроллеров/сервисов). Неверная реализация контроля доступа — один из наиболее распространённых классов рисков (Broken Access Control), что должно быть явно учтено в архитектуре. citeturn0search15turn12view0

Роли (базовый набор, расширяемый):

- **Суперадмин**: полный доступ ко всем настройкам, пользователям, ролям, интеграциям, ключам, системным логам, кэшу, шаблонам, custom code.
- **Админ**: управление контентом, категориями, медиа, меню/навигацией, SEO‑настройками, публикацией; без системных «разрушительных» операций (например, смена master‑ключей, миграции).
- **Редактор**: создание/редактирование/публикация страниц и постов, управление категориями (опционально), запуск LLM‑генерации в рамках политики, доступ к предпросмотру.
- **Автор**: создание/редактирование собственных материалов, отправка на ревью, просмотр статуса, но без публикации (если включён workflow).
- **Наблюдатель (read-only)**: чтение админки/контента без изменений (для аналитики/аудита).

Сценарии использования (минимально необходимые):

1) Редактор создаёт блог‑пост в WYSIWYG, назначает категории, настраивает SEO‑атрибуты (meta title/description, canonical, robots), запускает LLM‑помощника для черновика/улучшения и публикует.

2) Редактор создаёт страницу из подготовленных блоков (текст, заголовки, списки, медиа, карусели и т. п.), добавляет блок «Листинг постов по категории», получает предпросмотр, публикует. Механика листинга должна быть SEO‑корректной: отдельные URL для страниц пагинации, индексируемость управляется; infinite scroll допускается только при наличии crawlable пагинированной версии (если такой режим включён). citeturn10search9turn10search15

3) Интеграция (CI/скрипт/бот) публикует или обновляет материалы через API по токену, без доступа к админским сессиям. Практика разделения read‑only Content API и привилегированного Admin API подтверждается подходами Ghost: Content API предназначен для доставки опубликованного контента, read‑only и «полностью кэшируемый», а Admin API — для управления/публикации и требует аутентификации токенами. citeturn5search5turn5search1

4) Суперадмин настраивает политики безопасности (CSP, допустимые домены медиа/скриптов), настраивает кэширование и мониторинг, управляет ключами интеграций и ролями.

## Функциональные требования

### Контентная модель

В CMS должны быть следующие сущности (обязательные поля и поведение):

**Пост (Blog Post)**  
Назначение: публикация в блоге, индексируемая страница, поддержка категорий, RSS/Atom, разметка Article/BlogPosting (опционально).

Минимальные поля:
- id (UUID или bigint), title, slug, status (draft/review/scheduled/published/archived), created_at, updated_at, published_at, author_id.
- content_html (санитизированный HTML из WYSIWYG) + content_plain (для поиска/экспорта) + excerpt.
- categories[] (многие‑ко‑многим).
- featured_image (asset_id) + alt/caption/credits.
- SEO: meta_title, meta_description, canonical_url (опционально авто), robots_directives (index/noindex, follow/nofollow, noarchive, nosnippet и т. п. через meta robots или заголовок X‑Robots‑Tag). citeturn6search1turn6search12
- Structured data: JSON‑LD шаблон/генерация (опционально, но поддержка должна быть заложена).

**Страница (Page)**  
Назначение: SEO‑страница (landing/статья/категорийная/служебная), редактируется блоками, может включать динамические вставки (листинги постов), а также custom code (строго ограничено ролями/политиками).

Минимальные поля:
- id, title, slug, status, created_at, updated_at, published_at.
- content_blocks (JSON) — структурированное описание блоков страницы; серверный рендер в HTML.
- custom_code (опционально, с политиками и сегментацией: head_html, head_css, body_html, body_js или «вставки» с маркерами). Важно: любые функции «вставки произвольного HTML» потенциально обходят фильтры редактора; аналогично CKEditor подчёркивает, что функции HTML embed могут обходить механизмы фильтрации, поэтому серверная политика и ограничения обязательны. citeturn4search13
- SEO поля аналогично посту + page_type (например: “landing”, “about”, “category_index”, “utility”).

**Категория (Category)**  
Назначение: таксономия для постов и построения листингов.

Минимальные поля:
- id, title, slug, description (опционально), parent_id (опционально для иерархии), SEO поля (meta_title/description, robots, canonical), cover_image (опционально).

**Медиа‑объект (Asset)**  
Назначение: библиотека файлов (изображения/видео/документы), повторное использование, alt‑тексты и метаданные.

Поля:
- id, type, storage_path/url, mime_type, size, width/height (если применимо), created_at, checksum, alt, title, caption, credits/license (опционально).

**Пользователь / Роль / Permission**  
Назначение: RBAC, аудит действий, контроль доступа.

Поля:
- User: id, email/login, password_hash, status, created_at, last_login_at, 2FA (опционально).
- Role: id, name, permissions[].
- Audit log: actor_id, action, entity_type, entity_id, timestamp, ip, user_agent.

Пароли должны храниться с корректным хэшированием (например, Argon2id через стандартные средства PHP) и соответствовать практикам безопасного хранения. PHP поддерживает PASSWORD_ARGON2ID через password_hash при наличии сборки с Argon2. citeturn9search2turn9search1

### Редакторы и блоки

**WYSIWYG редактор для постов**:
- Встроенный редактор с базовым форматированием, ссылками, списками, цитатами, вставкой медиа, таблицами (опционально).
- Генерируемый HTML должен проходить серверную санацию (whitelist допустимых тегов/атрибутов + контекстный output encoding при рендере). OWASP подчёркивает, что XSS — критичный класс атак и защита требует корректной стратегии (экранирование/санитизация/ограничение). citeturn4search0turn9search16  
- Нельзя полагаться только на клиентские фильтры: например, CKEditor прямо указывает, что редактор **не является полноценным XSS‑фильтром** и рекомендует явно разрешать нужную разметку, а не «всё подряд». citeturn4search1  
- Для редакторов класса TinyMCE типовой подход — санация DOM через специализированные инструменты (например, DOMPurify), но серверная проверка остаётся обязательной, т. к. клиент — недоверенная среда. citeturn4search2

**Редактор страниц на блоках**:
- Визуальная сборка страницы из заготовленных блоков (минимум): Заголовок, Текст (rich text), Изображение, Видео (embed по URL), Галерея/Карусель, Список (ul/ol), Разделитель, CTA‑кнопка, Таблица (опционально), HTML‑вставка (restricted), Блок «Листинг постов» (по категории/тегу/фильтрам), Блок «FAQ» (для разметки FAQPage, опционально).
- Блоки должны храниться в структурированном формате (JSON schema блока + атрибуты + контент). Практика блочной модульности является современным стандартом в CMS‑строителях; например, WordPress описывает, что блоки хранятся в БД или шаблонах в HTML‑совместимом виде с чёткими разделителями (на базе HTML‑комментариев), чтобы разметка оставалась валидной HTML. citeturn5search0turn5search11  
- Серверный рендер блоков обязателен для SEO (HTML должен быть доступен без выполнения JS, иначе возникают риски неполной индексации листингов/контента).

**Вставка custom code на страницах**:
- Доступ только ролям Admin/SuperAdmin (по умолчанию).
- Политики:  
  - режим “safe embed” (разрешён только заранее определённый набор встраиваний: YouTube/Vimeo/карты/виджеты по whitelist доменов, без произвольного JS);  
  - режим “advanced” (произвольный HTML/CSS/JS) — включается явно и сопровождается предупреждениями и требованиями к CSP.
- При включении custom JS система должна поддерживать настройку Content Security Policy как слоя защиты от XSS/инъекций, что прямо описывается как основная цель CSP (контроль загружаемых ресурсов и смягчение XSS). citeturn5search3turn5search7

### SEO-функции “из коробки”

**URL и каноникализация**:
- ЧПУ (slug‑based), генерация URL по правилам:  
  - Посты: `/blog/<slug>` или `/blog/<yyyy>/<mm>/<slug>` (конфигурируемо).  
  - Страницы: `/<slug>` (поддержка вложенных страниц `/<parent>/<child>` опционально).  
- Автогенерация canonical URL и возможность ручного переопределения. Google описывает rel=canonical как один из способов сигнализировать preferred URL для дублей/похожих страниц. citeturn0search0turn0search4  
- Автоматические 301‑редиректы при смене slug (хранить историю URL). Для SEO необходимо избегать множества активных дублей и поддерживать консолидацию сигналов.

**Robots directives и index management**:
- Управление индексируемостью: для каждой страницы/поста/категории → meta robots или X‑Robots‑Tag. Спецификация robots meta и X‑Robots‑Tag описана Google. citeturn6search1  
- Реализация `noindex` должна быть корректной: Google показывает базовый паттерн `<meta name="robots" content="noindex">` и отдельно подчёркивает, что robots.txt **не** предназначен для скрытия страниц из индекса (он — про crawl access; для исключения из индекса нужен noindex или авторизация). citeturn6search5turn6search0

**Sitemap и RSS/Atom**:
- Автогенерация `sitemap.xml` и (при необходимости) `sitemap_index.xml` с разбиением по типам контента и размеру. Google описывает форматы и принципы построения и отправки sitemap, а также то, что sitemap помогает поисковику эффективнее обходить сайт. citeturn0search1turn0search13  
- Корректный `lastmod` (обновлять при изменении материала).  
- RSS/Atom фиды для блога (минимум: общий фид, фид по категории). Практика использования sitemap и RSS/Atom как источников обновлений описывается в рекомендациях Google. citeturn0search5

**Structured data (JSON‑LD)**:
- Поддержка генерации JSON‑LD для: Article/BlogPosting (пост), WebPage (страница), BreadcrumbList (хлебные крошки), Organization/WebSite (сайт), FAQPage (если есть FAQ‑блок).  
- Соблюдение general structured data guidelines и policies, чтобы не нарушать требования к eligibility rich results. citeturn0search2  
- Предпочтительный формат structured data — JSON‑LD (Google рекомендует его как более простой в поддержке и менее склонный к ошибкам). citeturn0search6

**Мультиязычность (опционально, но заложить в архитектуру)**:
- Поддержка разных URL для разных языков и hreflang annotations. Google рекомендует разные URL для языковых версий и использование hreflang для корректной привязки локализованных вариантов. citeturn6search19turn6search2

### Листинги и категории

- Категории должны связываться с постами (многие‑ко‑многим).
- В редакторе страниц должен существовать блок «Листинг», который может:
  - выводить последние посты;
  - выводить посты конкретной категории/набора категорий;
  - фильтровать по автору, статусу (только published на фронте), дате;
  - поддерживать сортировку (новые/популярные — популярность опционально).
- Пагинация: отдельные URL страниц листинга (`?page=2` или `/page/2`), серверный рендер ссылок.  
- Если используется infinite scroll/load‑more на фронте, система должна уметь генерировать crawlable «компонентные страницы»/пагинированную серию, чтобы поисковые системы могли обойти контент. citeturn10search15turn10search9  
- Учесть, что поддержка rel=next/prev в Google больше не актуальна (Google указывает, что rel=prev/next больше не поддерживается), поэтому SEO‑логика должна опираться на самостоятельную индексируемость страниц пагинации, внутреннюю перелинковку, canonical и качественный HTML контент. citeturn10search19

### API для публикации и доставки контента

Архитектурно реализовать два класса API:

**Content API (public, read-only)**  
Назначение: отдача опубликованного контента для фронтенда/мобильного приложения/интеграций, поддержка сильного кэширования. Ghost описывает Content API как read‑only и «fully cachable», доступ контролируется API key. citeturn5search5

Требования:
- GET‑эндпоинты: `/api/content/v1/posts`, `/api/content/v1/pages`, `/api/content/v1/categories`, `/api/content/v1/assets`.
- Фильтрация, сортировка, пагинация.
- Возврат только published.
- Заголовки кэширования: `Cache-Control`, `ETag`, `Last-Modified` (если возможно).
  - Поведение и управляющие заголовки описаны в стандарте HTTP caching (RFC 9111). citeturn3search2  
  - Семантика статус‑кодов и базовая модель HTTP — RFC 9110 (использовать корректные 200/304/404/429/500 и т. д.). citeturn3search3turn3search11
- API‑ключи: минимум “public key” для доступа (rate limit по ключу).

**Admin API (private, write)**  
Назначение: создание/редактирование/публикация контента, управление медиа и категориями, запуск LLM‑операций, управление интеграциями.

Требования:
- CRUD эндпоинты: `/api/admin/v1/posts`, `/api/admin/v1/pages`, `/api/admin/v1/categories`, `/api/admin/v1/assets`.
- Операции статуса: publish/unpublish/schedule.
- Валидация входных данных (OWASP рекомендует явные практики input validation как базовую защиту). citeturn4search4
- Аутентификация: токены интеграций (PAT) и/или JWT.
  - JWT как стандарт описан RFC 7519. citeturn1search3
- Скоупы токенов: `posts:write`, `pages:write`, `assets:write`, `llm:generate`, `settings:read` и т. п.
- Логирование действий (audit trail).

**Документирование API**:
- Спецификация OpenAPI 3.1, автогенерация документации и клиентов. OpenAPI Initiative описывает OAS как формальный стандарт описания HTTP API и пользу для понимания, генерации клиентов и тестов. citeturn1search13turn1search2

### LLM-интеграция для генерации страниц и контента

Функции LLM должны быть встроены как “assistive”, с явными ограничениями:

- Генерация черновиков постов и страниц:  
  - LLM возвращает **структурированный результат**, соответствующий схеме блоков (для страниц) или безопасному HTML‑подмножеству (для постов).  
- Генерация SEO‑полей: meta title/description, план статьи, FAQ‑блоки, предложения по внутренней перелинковке, черновик JSON‑LD.
- Генерация по шаблонам: «страница услуги», «гид/лонгрид», «категорийная страница», с конструктором из разрешённых блоков.
- Human‑in‑the‑loop: любой LLM‑результат сохраняется как draft и требует ручной проверки перед публикацией.

Безопасность LLM‑интеграции — обязательная часть ТЗ:
- В OWASP Top 10 for LLM Applications prompt injection указывается как ключевой риск; также выделяются insecure output handling, model DoS и др. citeturn1search0turn1search1  
- entity["organization","National Cyber Security Centre","uk government agency"] в материале “Prompt injection is not SQL injection (it may be worse)” подчёркивает, что prompt injection отличается по природе и может быть «хуже» классических инъекций, поэтому систему нужно проектировать так, чтобы снижать последствия компрометированного вывода, а не рассчитывать на полное устранение класса атак. Автором указан entity["people","David C.","ncsc technical director"]. citeturn2search0  

Следствия для требований:
- Жёсткая изоляция: LLM **не получает** прямой доступ к секретам, токенам, БД, файловой системе.
- Если LLM формирует “действия” (например, создать страницу), то выполняется через строго типизированный «план» (JSON schema) и дополнительную валидацию/санитизацию.
- Лимиты: rate limit, ограничение размера промптов/контента, защита от model DoS (OWASP LLM risk list включает Model Denial of Service). citeturn1search0
- Политики данных: запрет отправки персональных данных/секретов в LLM без явного согласия и маскирования.

## Нефункциональные требования

### Производительность и кэширование

Основной принцип: максимальная доля трафика (анонимные пользователи) должна обслуживаться без тяжёлых операций (без лишних запросов к БД и без повторного рендера).

Требования:
- Поддержка opcode cache: включение и рекомендации по настройке PHP OPcache (ускоряет PHP за счёт хранения предкомпилированного байткода и устранения повторного парсинга на каждый запрос). citeturn2search5turn2search1  
- HTTP‑кэширование: корректные заголовки Cache-Control/ETag/Last‑Modified для публичного контента; поведение контролируется стандартом HTTP caching. citeturn3search2turn3search10  
- Full-page cache (уровень приложения): кэш HTML по URL для анонимных запросов, инвалидация при публикации/обновлении; обход кэша для авторизованных сессий.
- Поддержка серверного кэширования на уровне веб‑сервера как опции: при использовании Nginx возможно задействовать fastcgi_cache (директивы описаны в документации модуля ngx_http_fastcgi_module). citeturn3search0turn3search8  
- Сжатие ответов: gzip/brotli (в зависимости от окружения), поскольку компрессия является стандартным способом повышения производительности по сети. citeturn7search2turn7search6  
- Оптимизация медиа: генерация responsive изображений (webp/avif где возможно), lazy loading, корректный alt. (Рекомендации по общим performance best practices для web включают оптимизацию изображений и сжатие). citeturn7search13  

Целевые метрики (как критерии приемки, измеряются для типовой темы CMS):
- TTFB (кэшированный HTML): < 200 мс на VPS‑классе; на shared — целевое “минимально возможное”.
- LCP/INP/CLS — “Good” по Core Web Vitals для ключевых шаблонов. Понимание роли CWV и их связи с поиском описывает Google. citeturn7search0turn7search4

## Архитектура и технологии

### Общая архитектура

Рекомендуемый стиль: **modular monolith** (ядро + модули), чтобы упростить деплой на хостинг и избежать микросервисной сложности.

Компоненты:
- Web UI (frontend rendering) — серверный рендер HTML по маршрутам.
- Admin UI — отдельная зона `/admin`, SPA‑lite или серверные шаблоны.
- API слой — Content API и Admin API (версионирование).
- Сервис блоков — schema + renderer + sanitizer.
- Сервис поиска — БД full‑text (минимум) + опционально внешний движок.
- Сервис кэширования — file cache по умолчанию + опционально Redis.
- Сервис LLM — абстрактный провайдер (внешний API), очередь/лимиты.

### Технологический стек (целевой)

- PHP: минимально 8.2+, предпочтительно 8.3/8.4 по срокам security support (планировать поддержку веток с учётом календаря php.net). citeturn8search0  
- Веб‑сервер: Apache (mod_rewrite) и/или Nginx.
- БД: MySQL/MariaDB (по умолчанию), PostgreSQL (опционально).
- Шаблоны: Twig/Blade или собственный lightweight renderer.
- Стандартизация HTTP слоя (опционально): PSR‑7 интерфейсы для HTTP‑сообщений (для лучшей тестируемости/совместимости библиотек). citeturn8search2  
- Описания API: OpenAPI 3.1. citeturn1search2turn1search13  

### Роутинг и динамические URL

Т. к. страницы и посты определяют URL через slug, система должна поддерживать динамическое разрешение маршрутов:
- На входе: попытка сопоставить URL с page/post/category.
- Кэш маршрутов: хранить “slug → entity” map в памяти/файле.
- При публикации/изменении slug: пересборка карты маршрутов и инвалидация кэша.

### Политики рендера и хранения контента

- Посты: хранить как sanitized HTML + извлечённый plain text.
- Страницы: хранить как JSON блоков.  
  - Renderer должен быть детерминированным (одна и та же структура даёт одинаковый HTML), что критично для кэша.
- Для блоковой модели можно ориентироваться на практику “валидный HTML при сериализации” (в WordPress блочная структура хранится в HTML‑совместимом виде с разделителями на базе комментариев), но в данном продукте предпочтительнее явный JSON для строгой типизации и LLM‑интеграции. citeturn5search0

## План разработки, тестирование и критерии приемки

### Этапность

**MVP (релиз 1)**:
- Контент: посты, страницы, категории, медиа.
- Редакторы: WYSIWYG для постов, блок‑редактор для страниц (ядро блоков).
- SEO‑ядро: canonical, meta robots, sitemap.xml, robots.txt генерация/шаблон, JSON‑LD базовый (Article/WebPage/Breadcrumb).
- Админка: RBAC, пользователи, аудит, предпросмотр, schedule publish.
- API: Content API (read-only) + Admin API (write) с токенами.
- Кэширование “из коробки”: HTML page cache + OPcache рекомендации + Cache-Control/ETag для Content API.
- Минимальная LLM‑интеграция: генерация черновика поста, генерация структуры страницы блоками, генерация SEO‑полей; всё только в draft.

**Релиз 2**:
- Версионность контента/ревизии UI.
- Расширенные блоки (FAQ, таблицы, CTA‑варианты, embed‑провайдеры).
- Мультиязычность + hreflang.
- Расширенный LLM: контент‑планы, пакетная генерация, “LLM‑помощник редактора”, контроль качества/тональности.
- Расширенные политики безопасности (настройки CSP в UI, whitelist embed).
